# 〇、需求整理与系统全景

## 0.1 角色与权限

| 角色 | 说明 | 核心权限 |
|---|---|---|
| 学生/校友 | 资料来源主体 | 提交/编辑本人资料（可选：仅由学院代录入） |
| 二级学院管理员 | 一级审核 | 审核本学院提交的资料（通过/驳回/退改） |
| 教务处管理员 | 二级审核 | 复审学院已通过资料，入库 |
| 校级管理员 | 资料库运营 | 资料维护、分类、上架/下架、轮播配置、账号管理 |
| 大屏展示端 | 展览馆一体机（无登录） | 公开浏览 + 人脸识别定位轮播 |

## 0.2 核心业务流（审核流转）

```
草稿 draft
  → 提交 → 待学院审核 pending_college
      → 学院通过 → 待教务处审核 pending_academic
                      → 教务处通过 → 已入库 approved
                                        → 校级上架 published（大屏可见）
                                        → 下架 unpublished
      → 学院驳回 / 教务处驳回 → rejected（含驳回意见，可退回修改）
```

## 0.3 人脸识别交互流（关键链路）

```
一体机摄像头采集 → Arcface 提取特征向量
  → 与 face_feature 库比对（阈值匹配）
    → 命中：定位该校友 → 大屏切换为「TA 的专属成长轨迹」轮播
    → 未命中：保持默认全库轮播 / 提示「未录入，欢迎采集」
```

> 注：Arcface 你已本地调通。建议 Arcface 单独作为一个 Python 推理服务（FastAPI），Spring Boot 通过 HTTP 调用，特征向量与比对结果回写 MySQL。这样 Java 端不耦合算法环境。

## 0.4 部署形态

- **管理后台**：PC 浏览器（学院/教务处/校管使用）
- **大屏端**：展览馆一体机，横屏（建议 1920×1080 或 4K），Chrome kiosk 全屏，挂载摄像头
